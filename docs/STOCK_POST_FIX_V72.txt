AIMS STOCK POST FIX V72
=======================

MASALAH
-------
Endpoint:
POST /api/v1/goods-receipts/{id}/post-scanned-to-stock

Sebelumnya gagal dengan PostgreSQL error:
42P10 - there is no unique or exclusion constraint matching the ON CONFLICT specification

AKAR MASALAH
------------
Database memiliki partial unique index:

CREATE UNIQUE INDEX cost_movements_stock_movement_unique_idx
ON inventory.cost_movements(stock_movement_id)
WHERE stock_movement_id IS NOT NULL;

Backend sebelumnya menjalankan:

ON CONFLICT(stock_movement_id) DO NOTHING

PostgreSQL tidak dapat menggunakan partial unique index tersebut karena conflict target tidak menyertakan predicate index.

PERBAIKAN BACKEND
-----------------
Query di internal/shared/costledger/ledger.go diubah menjadi:

ON CONFLICT (stock_movement_id)
WHERE stock_movement_id IS NOT NULL
DO NOTHING

Dengan demikian query sesuai dengan index database yang sudah ada.

PERUBAHAN ERROR HANDLING
------------------------
Handler PostScannedGoodsReceiptToStock sekarang memakai response.Handle.

Dampaknya:
- domain/validation error tetap memakai HTTP status dan code yang tepat;
- error database tak terduga menjadi HTTP 500, bukan salah diklasifikasikan sebagai 400;
- detail internal database tidak diekspos pada production.

DATABASE
--------
Tidak diperlukan ALTER TABLE atau migration baru.
Schema database backup(15) sudah benar dan sudah memiliki index yang diperlukan.

STATUS GOODS RECEIPT
--------------------
Status berikut memang merupakan tahap yang berbeda dan aman:

ACCEPTED / PARTIAL_ACCEPTED / REJECTED
    Hasil penerimaan fisik saat dokumen dibuat.

CHECKED
    Barang sudah diperiksa dan siap masuk workspace Scan Masuk.

POSTED_TO_STOCK
    Seluruh SERIAL/QTY yang diterima sudah berhasil diposting ke saldo stok.

Flow backend yang berlaku:
ACCEPTED / PARTIAL_ACCEPTED / REJECTED -> CHECKED -> POSTED_TO_STOCK

Database constraint, backend state machine, dan frontend Scan Masuk sudah selaras.

KEAMANAN TRANSAKSI GAGAL
------------------------
Proses post menggunakan satu PostgreSQL transaction.
Error sebelum COMMIT akan me-rollback:
- stock_movements;
- stock_balances;
- cost_movements;
- cost_balances;
- perubahan item_units menjadi AVAILABLE;
- perubahan Goods Receipt menjadi POSTED_TO_STOCK.

Karena itu request yang gagal dengan 42P10 aman untuk dicoba ulang setelah backend baru dideploy.
Draft scan frontend juga tetap tersimpan.

VALIDASI SETELAH DEPLOY
-----------------------
1. Deploy backend V72.
2. Tidak perlu generate QR ulang.
3. Buka kembali Scan Masuk untuk receipt yang sama.
4. Pastikan 3/3 QR masih tampil.
5. Tekan Post ke Stok lagi.
6. Pastikan response 200 dan status menjadi POSTED_TO_STOCK.

Query pemeriksaan opsional:

SELECT receipt_id, receipt_no, status, posted_at
FROM transaction.goods_receipts
WHERE receipt_id = 1;

SELECT movement_id, movement_type, qty, ref_type, ref_id, ref_line_id
FROM inventory.stock_movements
WHERE ref_type = 'GOODS_RECEIPT' AND ref_id = 1;

SELECT unit_id, qr_code, status, current_warehouse_id
FROM inventory.item_units
WHERE source_type = 'GOODS_RECEIPT' AND source_id = 1
ORDER BY unit_id;

SELECT cost_movement_id, stock_movement_id, qty, unit_cost, total_cost
FROM inventory.cost_movements
WHERE ref_type = 'GOODS_RECEIPT' AND ref_id = 1;
